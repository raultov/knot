# Knot CLI Agent Skill

## Overview

This skill teaches AI agents how to use the **knot CLI** to search, explore, and analyze indexed codebases. The knot CLI provides the exact same capabilities as the knot MCP server, but through terminal commands instead of MCP protocol.

## Prerequisites

- `knot` CLI binary must be installed and accessible in PATH
- A knot index must be initialized in the target repository (via `knot-indexer`)
- Environment variables configured:
  - `QDRANT_URL` (default: http://localhost:6333)
  - `QDRANT_COLLECTION` (default: knot)
  - `NEO4J_URI` (default: bolt://localhost:7687)
  - `NEO4J_USER` (default: neo4j)
  - `NEO4J_PASSWORD` (required)

## Core Commands

### 1. Search for Code by Semantic Meaning

**Command:** `knot search "<query>" [--max-results <N>] [--repo <name>]`

**Purpose:** Find code entities by semantic meaning. This is your primary tool for exploratory searches.

**Parameters:**
- `<query>` (required): Natural language description of what you're looking for
  - Examples: "user authentication", "error handling", "database connection", "API routes"
- `--max-results <N>`: Limit number of results (default: 5, max: 20)
- `--repo <name>`: Filter to specific repository (optional, omit to search all)

**Output:** Markdown formatted results with:
- Entity names and file locations
- Function/method signatures
- Documentation and comments
- Related dependencies and usage patterns

**Example:**
```bash
knot search "authenticate user with JWT" --max-results 10 --repo my-api-backend
```

**When to use:**
- Finding code that handles a specific feature
- Searching for patterns (e.g., "caching strategy")
- Locating architectural patterns
- Discovery phase when you don't know exact names

---

### 2. Find All References to an Entity (Reverse Dependency)

**Command:** `knot callers "<EntityName>" [--repo <name>]`

**Purpose:** Comprehensive reverse dependency lookup. Answers: "Who uses this? What will break if I change this?"

**Parameters:**
- `<EntityName>` (required): Name of class, interface, function, or method
  - Supports partial names and signature fragments
  - Examples: "AuthService", "handleRequest", "processPayment"
- `--repo <name>`: Filter to specific repository (optional)

**Output:** References grouped by type:
- **Calls** — Function/method invocations
- **Extends** — Class inheritance (who extends this class?)
- **Implements** — Interface implementation (who implements this interface?)
- **References** — Type usage in annotations/signatures

Each reference shows:
- Entity name and kind (method, class, etc.)
- File path and line number
- Method signature if applicable

**Example:**
```bash
knot callers "UserService" --repo my-api
```

**When to use:**
- Impact analysis before refactoring
- Dead code detection (0 references = unused)
- Finding all tests for a feature
- Understanding dependency chains
- Refactoring safety checks

---

### 3. Explore All Entities in a File

**Command:** `knot explore "<file_path>" [--repo <name>]`

**Purpose:** List all classes, methods, functions, and properties in a file. Understand file structure at a glance.

**Parameters:**
- `<file_path>` (required): Absolute or relative path to source file
  - Examples: "src/main/java/AuthHandler.java", "src/services/user.ts"
- `--repo <name>`: Filter to specific repository (optional)

**Output:** File anatomy organized by entity type:
- Classes and interfaces
- Singleton objects (Kotlin)
- Companion objects (Kotlin)
- Methods and functions
- Properties (Kotlin)

Each entity shows:
- Name and line number
- Decorators/annotations
- Method signature
- First line of documentation

**Example:**
```bash
knot explore "src/handlers/auth.ts" --repo my-app
```

**When to use:**
- Quick overview of a file's structure
- Finding all methods in a class
- Understanding file organization before diving into code
- Navigating large modules

## Workflow Patterns

### Pattern 1: Feature Discovery

1. **Start with semantic search:**
   ```bash
   knot search "user login flow" --max-results 10
   ```

2. **Explore identified files:**
   ```bash
   knot explore "src/auth/login.ts" --repo my-app
   ```

3. **Find related code:**
   ```bash
   knot callers "loginUser" --repo my-app
   ```

### Pattern 2: Impact Analysis Before Refactoring

1. **Find the entity to refactor:**
   ```bash
   knot search "payment processor" --repo billing-service
   ```

2. **Find all dependencies:**
   ```bash
   knot callers "PaymentProcessor" --repo billing-service
   ```

3. **Explore each caller file:**
   ```bash
   knot explore "src/services/transaction.ts" --repo billing-service
   ```

### Pattern 3: Dead Code Detection

1. **Find specific method:**
   ```bash
   knot callers "legacyFunction"
   ```

2. **If output contains "No references found" → dead code**

3. **Verify by exploring the file:**
   ```bash
   knot explore "src/legacy/old-module.ts"
   ```

### Pattern 4: Understanding Architecture

1. **Search for architectural pattern:**
   ```bash
   knot search "dependency injection container" --max-results 15
   ```

2. **Find all uses of key interfaces:**
   ```bash
   knot callers "ServiceProvider"
   ```

3. **Explore implementation files:**
   ```bash
   knot explore "src/core/services.ts"
   ```

### Pattern 5: Cross-Language Analysis (Java + TypeScript)

1. **Find Java service:**
   ```bash
   knot search "user repository" --repo backend
   ```

2. **Find TypeScript API endpoints calling it:**
   ```bash
   knot callers "UserRepository" --repo backend
   ```

3. **Trace through both codebases iteratively**

## Output Interpretation Guide

### Search Results

```markdown
# Search Results for "authentication"

Found 3 entity(entities):

## Functions

- `authenticateUser` (line 42)
  - Signature: `async authenticateUser(email: string, password: string): Promise<User>`
  - Doc: Authenticates a user with email and password using bcrypt
```

**Interpretation:**
- Entity name, kind, and location
- Function signature shows parameters and return type
- Documentation tells you what it does
- Related entities show dependencies and who uses it

### Callers Results

```markdown
# References to `UserService`

Found 3 reference(s):

## Calls (1)

- **`getUserById`** (method) at `src/handlers/user.ts:25`
  - Signature: `async getUserById(id: string)`

## Extends (0)

## Implements (2)

- **`AdminService`** (class) at `src/admin/admin-service.ts:10`
```

**Interpretation:**
- Grouped by relationship type (Calls, Extends, Implements, References)
- Each reference shows who depends on the entity
- Count tells you importance (high count = critical entity)
- Line numbers let you navigate directly

### Explore Results

```markdown
# Entities in `src/auth/login.ts`

Found 5 entity/entities:

## Classes

- **`LoginHandler`** (line 10)
  - Signature: `export class LoginHandler`
  - Decorators: @Controller, @UseGuards

## Methods

- **`login`** (line 15)
  - Signature: `async login(email: string, password: string)`
```

**Interpretation:**
- Organized by entity type for easy scanning
- Line numbers enable quick navigation
- Decorators show important metadata
- Signatures show interface contracts

## Tips for Effective Usage

### 1. Always Start with Repository Filter
When working with multiple indexed repositories, specify `--repo` to avoid mixed results:
```bash
knot search "user service" --repo my-api  # Good
knot search "user service"                 # May return results from multiple repos
```

### 2. Use Semantic Search First
- Don't search for exact method names; use **what the code does**
- Good: `knot search "caching strategy for user data"`
- Bad: `knot search "Cache"`

### 3. Combine Commands for Deep Analysis
- Search → Explore → Callers is the typical flow
- This triangulation gives you full context

### 4. Interpret "No References Found"
- **Dead Code:** Zero callers + entity exists = unused
- **New API:** Zero callers + just created = expected
- **Static Method:** May be referenced via annotation or reflection (search for "Class.staticMethod")

### 5. Use Line Numbers for Navigation
- All outputs include file:line format
- You can jump directly to code in your editor

### 6. Check Multiple Results
- Don't stop at first search result
- Use `--max-results 10+` to see related implementations
- Different patterns may exist in different places

## Error Messages and Troubleshooting

### "No matching code found for your query"
**Cause:** Query is too specific or no index exists
**Solution:** 
- Try broader semantic query: "authentication" instead of "JWT token validation"
- Ensure index is up to date: `knot-indexer index <repo-path>`

### "No references found for `EntityName`"
**Possible interpretations:**
- Entity is truly unused (dead code)
- Entity was recently added (no callers yet)
- Search for different entity name (case-sensitive)

### Connection refused / Database errors
**Cause:** Qdrant or Neo4j not running
**Solution:**
- Check environment variables are set correctly
- Verify database services are running
- Check network connectivity

## Integration with AI Workflows

### For Code Analysis Tasks

```
Agent: "Analyze the authentication system"
Process:
1. knot search "authentication" --max-results 10
2. For each result: knot explore <file_path>
3. For each key entity: knot callers <entity>
4. Synthesize findings into analysis report
```

### For Refactoring Tasks

```
Agent: "Refactor PaymentService for testability"
Process:
1. knot explore "src/PaymentService.ts"
2. knot callers "PaymentService"
3. For each caller: knot explore <caller_file>
4. Plan refactoring based on dependency analysis
5. Verify after refactoring with knot callers again
```

### For Architecture Understanding

```
Agent: "Map the system architecture"
Process:
1. knot search "dependency injection" --max-results 15
2. knot search "event bus" --max-results 10
3. knot search "database layer" --max-results 10
4. knot callers "Container" (for DI)
5. Visualize relationships between modules
```

## Performance Notes

- **Search**: Fast (vector similarity in Qdrant) - typical response < 1 second
- **Callers**: Medium (graph traversal in Neo4j) - typical response < 2 seconds
- **Explore**: Very fast (direct file lookup) - typical response < 100ms

For very large codebases:
- Use `--max-results` to limit results
- Use `--repo` to filter to specific project
- Combine searches: narrow semantic search + targeted callers lookup

## When to Use Knot CLI vs MCP

| Scenario | Use CLI | Use MCP |
|----------|---------|---------|
| Terminal-only environments | ✅ | ❌ |
| IDE integration (with MCP support) | ❌ | ✅ |
| Bash scripting / automation | ✅ | ❌ |
| LLM-based code analysis tools | ✅ | ✅ |
| Quick ad-hoc exploration | ✅ | ❌ |
| Continuous integration pipelines | ✅ | ❌ |

Both tools access the **same databases** and provide **identical results** — choose based on your environment.
